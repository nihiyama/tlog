# Requirements To Systemization Guide

この skill は、曖昧な要求を「システムとして実装可能な要件」に変換する。

## Workflow

1. `AGENTS.md` を読み、プロジェクト目的・対象提供形態・責務境界・禁止事項を要約する。
2. `README.md` を読み、製品価値・主機能・利用文脈を要約する。
3. 要件整理の前提を定義する。
   - 対象ユーザー
   - 対象業務
   - 成功指標
   - 制約条件
4. ペルソナを 1-3 人作る。
   - 主ペルソナを 1 人定義する。
   - 必要なら副ペルソナを追加する。
   - 役割、課題、現行業務、意思決定基準を明示する。
5. ビジネス要件を整理する。
   - 解決したい課題
   - 期待効果（定量・定性）
   - スコープ内/外
   - 優先度
6. 業務仕様を整理する。
   - ユースケース
   - 業務フロー
   - 入出力
   - 例外処理
7. システム化要件に変換する。
   - 機能要件
   - 非機能要件
   - データ要件
   - インターフェース要件
   - 運用要件
8. AGENTS.md の責務境界へマッピングする。
   - VS Code 拡張
   - CLI
   - MCP
   - 共通ロジック
9. 不確実性を明示する。
   - 未確定事項
   - 依存関係
   - 追加ヒアリング項目

## Output Rules

- 出力は次の順番で構成する。
  1. 前提とコンテキスト
  2. ペルソナ
  3. ビジネス要件
  4. 業務仕様
  5. システム化要件
  6. 責務境界マッピング
  7. 未確定事項と次アクション
- 可能な限り表形式で示し、優先度は `Must / Should / Could` で付与する。
- 非機能要件は最低でも `性能・信頼性・セキュリティ・監査性・運用性` を確認する。
- セキュリティを脅かす提案や破壊的運用は避ける。
- 仕様や設計に関するドキュメントは `steering/` ディレクトリに配置する。

## Templates

詳細テンプレートは `references/requirement_templates.md` を使う。
必要に応じて `references/interview_questions.md` の質問集を使って不足情報を埋める。

## Quality Checklist

- ペルソナの課題と要件が対応している。
- ビジネス要件が機能要件にトレースできる。
- 非機能要件が測定可能な表現になっている。
- AGENTS.md の責務境界に矛盾しない。
- スコープ外が明示されている。
